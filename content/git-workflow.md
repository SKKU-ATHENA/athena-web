## 브랜치 전략

```
main          ← 안정 브랜치 (직접 push 금지)
├── feature/* ← 기능 개발
├── fix/*     ← 버그 수정
└── docs/*    ← 문서 작업
```

### 작업 흐름

1. `main`에서 새 브랜치 생성
2. 작업 후 커밋
3. PR 생성
4. CI 통과 확인
5. Merge

## 커밋 컨벤션

**한국어 동사형**으로 작성합니다.

```
<타입>: <설명>

feat: 지식 그래프 검색 API 추가
fix: 임베딩 차원 불일치 수정
docs: M1 회의록 작성
refactor: LLM 프로바이더 추상화 개선
chore: 의존성 업데이트
test: KG 추출 파이프라인 테스트 추가
```

### 타입 설명

| 타입 | 설명 |
|------|------|
| `feat` | 새로운 기능 |
| `fix` | 버그 수정 |
| `docs` | 문서 변경 |
| `refactor` | 리팩토링 |
| `test` | 테스트 추가/수정 |
| `chore` | 빌드, CI, 의존성 등 |

## PR 규칙

- 제목은 커밋 메시지와 동일한 형식
- 변경 사항 설명 필수
- 리뷰는 선택 (비전공 팀원 고려)
- **Merge 전 CI 통과 필수**
